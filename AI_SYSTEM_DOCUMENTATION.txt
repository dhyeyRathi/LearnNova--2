1. LLM INTEGRATION OVERVIEW
================================================================================

The LearnNova LMS integrates Google's Gemini API as its core AI engine powering
the "Nova" AI learning assistant. Nova is a context-aware chatbot that:

  - Provides intelligent learning recommendations
  - Generates personalized quiz feedback
  - Tracks student progress and achievements
  - Suggests next learning steps
  - Summarizes course content
  - Gives encouragement and motivation

KEY INTEGRATION FILES:
  ├─ src/utils/geminiService.ts      (Gemini API connection & initialization)
  ├─ src/utils/novaAgent.ts           (Tool detection & execution orchestration)
  ├─ src/utils/novaTools.ts           (Tool definitions & implementations)
  ├─ src/app/components/AIAssistant.tsx (Nova chat UI component)
  └─ src/app/context/AuthContext.tsx  (User context passed to tools)

3. NOVA AGENT SYSTEM ARCHITECTURE
================================================================================

The Nova Agent uses a tool-aware decision system that determines WHEN to use
tools vs. when to respond with pure text generation.

ARCHITECTURE FLOW:
  User Input → Tool Detection → Decision → Tool Execution → Response Generation

FILE: src/utils/novaAgent.ts

CORE FUNCTIONS:

  1) detectToolUsage(userMessage: string)
     Purpose: Determine if a tool should be called
     Process:
       - Analyzes user message for intent keywords
       - Sends to Gemini with TOOL_DETECTION_PROMPT
       - Gemini returns JSON with tool decision
       - Parses JSON response safely
     Output: { should_use_tool: boolean, tool_name?: string, tool_input?: any }
     
     TOOL DETECTION PROMPT TELLS GEMINI:
       "If user asks about: points/badge/progress → Use get_learner_progress"
       "If user asks about: quiz results/score → Use give_quiz_feedback"
       "If user asks about: course summary/overview → Use summarize_course"
       "If user asks about: what's next/next lesson → Use recommend_next_lesson"
       "Otherwise, respond naturally without tools"

  2) executeTool(toolName: string, toolInput: any, currentUser?: any)
     Purpose: Execute a selected tool with validated inputs
     Process:
       - Finds tool by name from tools array
       - Validates input against Zod schema
       - If no learner_id provided, uses currentUser.id
       - Executes tool's handler function
       - Returns ToolResult { success: boolean, data?: any, error?: string }

  3) processWithTools(userMessage: string, currentUser?: any)
     Purpose: Orchestrate the full tool-aware workflow
     Process:
       STEP 1: Call detectToolUsage() → Gemini decides if tool needed
       STEP 2: If NO tool needed → Return Gemini's natural response
       STEP 3: If TOOL NEEDED:
         - Call executeTool() with user context
         - Get tool result
         - Send tool result back to Gemini with context prompt
         - Gemini generates friendly response based on results
   
  4) toolAwareChatResponse(userMessage: string, currentUser?: any)
     Purpose: Main function called by UI components
     Returns: Promise<string> → Final AI response ready for display
     Accepts: currentUser (from AuthContext) to provide real data

4. AVAILABLE TOOLS & FUNCTIONS
================================================================================

FILE: src/utils/novaTools.ts

TOOL #1: summarize_course
────────────────────────────
Input Schema:
  {
    course_id: string (e.g., "course-001"),
    lesson_ids?: string[] (optional: specific lessons to summarize)
  }

Execution: async function summarize_course(input: CourseSummaryInput)

Logic:
  - Looks up course from mockCourses database
  - Retrieves course title, description, objectives
  - If lesson_ids provided, includes those lesson summaries
  - Returns formatted course overview with key learning outcomes

Use Cases:
  User: "What is this course about?" → Tool triggers
  User: "Give me a summary of JavaScript course" → Tool triggers
  User: "Tell me about course-001" → Tool triggers

Output:
  {
    success: true,
    data: {
      course_id: string,
      title: string,
      description: string,
      objectives: string[],
      lesson_count: number,
      estimated_duration: string
    }
  }


TOOL #2: give_quiz_feedback
────────────────────────────
Input Schema:
  {
    quiz_id: string,
    attempt_number: number,
    answers: [
      {
        question: string,
        selected: string,
        correct: string,
        is_correct: boolean
      }
    ]
  }

Execution: async function give_quiz_feedback(input: QuizFeedbackInput)

Logic:
  - Analyzes all answers in the quiz attempt
  - Identifies weak areas (questions marked is_correct = false)
  - Calculates score and performance percentage
  - Generates personalized feedback for improvement areas
  - Provides encouragement and specific study recommendations

Triggered When:
  - User completes a quiz in lesson player
  - QuizFeedbackModal calls this tool
  - Learner asks: "How did I do?" after quiz

Output:
  {
    success: true,
    data: {
      score: number (0-100),
      total_questions: number,
      correct_answers: number,
      performance_level: "excellent" | "good" | "needs-improvement",
      weak_areas: string[],
      strengths: string[],
      recommendations: string[],
      personalized_feedback: string
    }
  }


TOOL #3: get_learner_progress
──────────────────────────────
Input Schema:
  {
    learner_id: string (e.g., "learner-001" or current user id)
  }

Execution: async function get_learner_progress(input: LearnerProgressInput, currentUser?: any)

IMPORTANT: This tool now fetches REAL USER DATA when currentUser is provided!

Logic (WITH REAL DATA):
  - If currentUser provided: Uses currentUser.points from auth context
  - Calculates badge level based on points:
    • 0-99 points → "Newbie"
    • 100-299 points → "Learner"
    • 300-599 points → "Scholar"
    • 600-999 points → "Expert"
    • 1000+ points → "Master"
  - Determines points needed for next badge
  - Returns accurate progress data

Fallback (Mock Data):
  - If currentUser not provided, uses mockLearners data
  - Used for demo/testing purposes

Triggered When:
  User: "How many points do I have?" → Tool triggers
  User: "What's my current badge?" → Tool triggers
  User: "Show my progress" → Tool triggers
  User: "What do I need for the next level?" → Tool triggers

Output:
  {
    success: true,
    data: {
      points_total: number,        /* REAL: from user.points */
      current_badge: string,      /* Calculated from points */
      lessons_completed: number,
      points_needed_for_next_badge: number,
      next_badge: string,
      progress_percentage: number (0-100),
      recent_quizzes: array,
      streak_days: number
    }
  }


TOOL #4: recommend_next_lesson
───────────────────────────────
Input Schema:
  {
    learner_id: string,
    course_id: string
  }

Execution: async function recommend_next_lesson(input: RecommendLessonInput)

Logic:
  - Looks up course to find all lessons
  - Calculates learner's completion progress
  - Identifies next incomplete lesson
  - Returns structured recommendation with difficulty estimate
  - Suggests related resources or prerequisites

Triggered When:
  User: "What should I learn next?" → Tool triggers
  User: "What's the next lesson?" → Tool triggers
  User: "Help me continue where I left off" → Tool triggers

Output:
  {
    success: true,
    data: {
      recommended_lesson: {
        lesson_id: string,
        title: string,
        description: string,
        duration: string,
        difficulty: "beginner" | "intermediate" | "advanced",
        topics: string[]
      },
      progress_summary: string,
      motivation_message: string
    }
  }

8. ERROR HANDLING & FALLBACKS
================================================================================

ERROR SCENARIO 1: Missing API Key
────────────────────────────────────
Where: initializeChat()
Detection: typeof process.env.VITE_GOOGLE_AI_KEY === 'undefined'
User Message: "Gemini API key not configured"
Fallback: Returns mock response

ERROR SCENARIO 2: Network Error During Tool Detection
──────────────────────────────────────────────────────
Where: detectToolUsage() → getChatResponse()
Detection: Catch block in getChatResponse()
Behavior: Returns { should_use_tool: false, response: "I encountered an error..." }
Result: User sees error message, can retry

ERROR SCENARIO 3: Invalid Tool Input
──────────────────────────────────────
Where: executeTool() → validation.success check
Detection: Zod schema validation fails
Response: { success: false, error: "Invalid input: ..." }
User Message: "I encountered an error: Invalid input..."

ERROR SCENARIO 4: Tool Execution Fails
────────────────────────────────────────
Where: catch block in tool function
Detection: Unexpected error during execution
Response: { success: false, error: String(error) }
User Message: "I encountered an error. Please try again."

ERROR SCENARIO 5: Tool Not Found
──────────────────────────────────
Where: executeTool() → tool lookup
Detection: tools.find() returns undefined
Response: { success: false, error: "Tool {toolName} not found" }

FALLBACK MECHANISMS:
  1. If API Key Missing → Use text-based responses only
  2. If Network Error → Show "Connection issue, trying again..."
  3. If Tool Failed → Provide general helpful response
  4. If JSON Invalid → Assume no tool needed, continue with text
